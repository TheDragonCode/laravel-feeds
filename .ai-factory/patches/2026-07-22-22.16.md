# Feed benchmarks compared unrelated runner performance

**Date:** 2026-07-22 22:16
**Files:** `.github/workflows/tests.yml`, `.gitignore`, `tests/Benchmark/FeedGenerationTest.php`, `tests/Benchmark/assert-regression.php`, `tests/Datasets/benchmark.php`, `tests/Helpers/Benchmark/BenchmarkRegression.php`, `tests/Helpers/Benchmark/RegressionThresholds.php`, `tests/Unit/Benchmark/BenchmarkRegressionTest.php`, `tests/.benchmarks`
**Severity:** medium

## Problem

The benchmark job intermittently failed unchanged feed generation code. Identical commits passed and failed on separate GitHub-hosted runners, and raising the allowed regression percentages did not make the job stable.

## Root Cause

The job compared absolute execution times from the current hosted runner with snapshots captured once on a different machine. Trimming individual iterations reduced short spikes but could not remove systematic CPU and filesystem differences between runners. The committed snapshots also became stale as the measured generation path changed.

## Solution

The workflow now measures the base and candidate production code on the same pinned Ubuntu, PHP, Composer, dependency installation, and benchmark harness. Four runs are collected in alternating base/candidate order, and median times are compared using the original format-specific limits. Absolute committed snapshots were removed. A deterministic regression test and an end-to-end temporary slowdown check confirm that an intentional regression above the limit still fails.

## Prevention

- Compare performance candidates with their base revision on the same runner and dependency installation.
- Alternate measurement order and aggregate multiple runs to reduce temporal runner drift.
- Keep benchmark setup outside the measured callback and store only generation-time results.
- Verify the comparison layer with deterministic noisy samples and an intentional slowdown above every configured limit.
- Do not relax thresholds until runner variance and stale baselines have been ruled out.
